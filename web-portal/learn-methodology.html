The 7-Phase Workflow

From absolute specifications to verified compliance. Explore the rigorous development lifecycle that governs every project build cycle.

Phase 1

Specify (Π.1.1)

"What are we building, and why does it matter?"

Before writing code, we write behavioral specifications. We define user scenarios, outline requirements, and list all technical constraints. The specification is written in standard markdown format under `specs/` or as a `specs.md` file.

Key Outputs:

spec.md User Stories Acceptance Criteria
📝
Phase 2

Clarify (Π.1.1a)

"Are there any gaps or ambiguities in our intent?"

If the system specification is ambiguous, the program cannot be verified. We perform clarification cycles to address gaps, list questions, and confirm requirements with stakeholders, locking down the specification contract.

Key Outputs:

Clarification Q&A Locked spec.md
💬
Phase 3

Plan (Π.3.1, Π.4.1)

"How do we decompose our specification into architecture?"

We design the module architecture, choose database and external integrations, establish APIs, and document all significant decisions in Architecture Decision Records (ADRs). ADRs must strictly trace back to axioms.

Key Outputs:

plan.md ADRs (docs/decisions/) API Contracts
🗺️
Phase 4

Tasks (Π.1.2a)

"What is the optimal path to build the system?"

We decompose our plan into small, atomic tasks, organizing them logically in a `tasks.md` checklist. We define strict dependencies, building from the foundation (infrastructure/models) upward to services and endpoints.

Key Outputs:

tasks.md Dependency Trees
📋
Phase 5

Implement (Π.2.1, Π.2.2)

"Write clean, deterministic, test-aligned code."

We implement features following a Test-Driven Development (TDD) cycle. We write failing unit/integration tests matching the specification requirements, write code that implements the feature to pass tests, and refactor for quality.

Key Outputs:

Source Code Unit & Integration Tests Atomic Commits
💻
Phase 6

Verify (Π.5.1b)

"Does our code strictly satisfy the spec.md contracts?"

We run local compliance checks using the `ade-compliance` engine. We check if specifications exist, if all files have deterministic tests, and if all implementation files have traceability comments that link to specs and axioms.

Key Outputs:

ade-compliance Report Test Coverage Matrix
🧪
Phase 7

Analyze (Π.3.1)

"How can we optimize our principles and workflow?"

A post-implementation retrospective. We evaluate any compliance failures, calculate human review rates, verify audit chain logs, and update our organizational templates to refine future development iterations.

Key Outputs:

Post-Mortem / Retro Knowledge Updates
📊
